# CineMatch — Full Build Specification

## 1. What You Are Building

CineMatch is an AI agent that recommends movies personalized to a user's taste and verified as available to stream/rent/buy on a specific platform in a specific country. It solves two concrete problems that general-purpose LLMs fail at: (1) personalization — LLMs default to the most popular titles and ignore nuance, and (2) availability — LLMs have no real-time knowledge of streaming catalogs and hallucinate availability.

The agent uses a RAG pipeline over an IMDB movie dataset (Pinecone) for taste-matching, and the TMDB API for live availability verification. It never asserts a movie is available without a live TMDB lookup. Scope is movies only — no TV shows.

## 6. Backend — Agent Pipeline

### Overview

The pipeline is managed by a **deterministic orchestrator** (`agent/orchestrator.py`). It is plain Python control flow — no LLM decides the top-level routing. The orchestrator calls each stage in sequence and handles the ReAct ↔ Reflection retry loop.

```python
# Pseudocode — orchestrator logic
async def run_pipeline(prompt: str, conversation_history: list, steps: list) -> str:
    # Stage 0 — Taste Extraction
    # If the input is already structured (matches the UserPreferences schema), skip
    # the LLM call and map the fields directly. Otherwise, make one LLM call.
    preferences = await taste_extractor(prompt, conversation_history, steps)

    # Stage 1+2 — ReAct ↔ Reflection loop
    react_context = ReactContext(preferences=preferences, feedback=None, use_fallback=False)

    for pass_num in range(1, config.MAX_PASSES + 1):
        # Stage 1 — ReAct Agent (agentic tool-calling loop)
        draft = await react_agent(react_context, steps)

        # If ReAct triggered ask_user_clarification, short-circuit
        if draft.is_clarification:
            return draft.clarification_question

        # Stage 2 — Reflection Agent (critique + verify)
        verdict = await reflection_agent(draft, preferences, steps)

        # If Reflection triggered ask_user_clarification, short-circuit
        if verdict.decision == "clarify":
            return verdict.question

        if verdict.decision == "approve":
            return verdict.final_response

        # Reject — feed critique back to ReAct for next pass
        react_context.feedback = verdict.critique
        react_context.use_fallback = verdict.use_fallback  # bool

    # Exhausted passes — return best effort from last draft
    return compose_best_effort(draft, preferences)
```

### Stage 0 — Taste Extractor (`agent/taste_extractor.py`)

**One LLM call** (or zero if the input is already structured — see the bypass note below).

**System prompt direction:** "You parse a natural-language movie recommendation request into structured JSON. Extract these fields: `mood` (free text describing vibe/feel), `genres` (list of genre strings), `similar_to` (list of movie title strings the user wants something like), `exclude` (list of movie title strings the user has seen or wants to exclude), `country` (string — the country for availability checking), `platforms` (list of streaming platform strings), `year_min` (int or null), `year_max` (int or null), `min_rating` (float or null). If conversation_history is provided, fold in previously stated preferences that aren't overridden by the current message. If a required field (country, platforms) is genuinely missing and can't be inferred, set it to null — the downstream agent will handle asking the user. Return valid JSON only, no prose."

**Input:** The user's prompt + conversation_history (if any).
**Output:** A Pydantic model `UserPreferences` with the fields above.

The LLM is called with `response_format={"type": "json_object"}` to guarantee valid JSON. Parse the response into the Pydantic model. Log a step with module `TasteExtractor`.

**Structured-input bypass:** If the `/api/execute` request includes a flag, or the `prompt` is already a JSON object matching the `UserPreferences` schema, skip the LLM call and parse directly. This saves one LLM call whenever the caller already has structured preferences (e.g. a future UI form, or a direct API caller), and requires no frontend to exist for the backend to support it.

### Stage 1 — ReAct Agent (`agent/react_agent.py`)

**True agentic loop** using the LLM's native function-calling (tool_use) capability. This is NOT a deterministic sequence — the LLM decides which tool to call, in what order, and when to stop.

**System prompt direction:** "You are CineMatch's retrieval agent. Your job is to find 1–5 movie recommendations matching the user's preferences. You have access to tools. Think step by step: consider the user's taste, decide what to search for, evaluate results, and build a draft candidate list. Every candidate must have a TMDB ID before you finish (this comes for free from `rag_search`'s metadata or from `tmdb_fallback_search` — you do not need to look it up separately). Availability is verified downstream by the Reflection stage, so you do not need to confirm it yourself unless you're using `tmdb_fallback_search`, which already returns availability-filtered results. Exclude any movie the user listed in their 'exclude' list. You should aim for a mix of well-known and lesser-known picks when possible. {If feedback from a previous Reflection pass is present, address it specifically.} {If use_fallback is true, you are allowed and encouraged to use tmdb_fallback_search in addition to rag_search.} When you're satisfied with your candidate list, return it as your final message in the specified JSON format."

**Tools available to ReAct** (defined as OpenAI function-calling tool schemas):

1. **`rag_search`** — see §7 for full spec
2. **`tmdb_fallback_search`** — see §7 (only callable when `use_fallback=True` in context; if `use_fallback=False`, do not include this tool in the tool list for that pass)
3. **`ask_user_clarification`** — see §7

**Agentic loop implementation:**

```python
async def react_agent(context: ReactContext, steps: list) -> ReActDraft:
    messages = build_react_messages(context)  # system prompt + user context
    tools = build_react_tools(context)  # conditional: include tmdb_fallback only if allowed
    
    for cycle in range(config.MAX_TOOL_CALLS):
        # Call LLM with messages and tools
        response = await llm_client.chat(messages, tools=tools)
        
        # Log the LLM call as a step
        log_step(steps, module="ReActAgent", ...)
        
        # If the LLM returned a text response (no tool calls), the draft is ready
        if not response.tool_calls:
            return parse_draft(response.content)
        
        # Process each tool call
        for tool_call in response.tool_calls:
            # Validate tool name against allowed set
            if tool_call.function.name not in ALLOWED_TOOLS:
                tool_result = f"Error: tool '{tool_call.function.name}' does not exist. Available tools: {list(ALLOWED_TOOLS)}. Try again."
            else:
                # Validate args schema
                try:
                    tool_result = await execute_tool(tool_call, context)
                except Exception as e:
                    tool_result = f"Error executing {tool_call.function.name}: {str(e)}"
            
            # Log the tool execution as a step
            log_step(steps, module=f"ReActAgent/{tool_call.function.name}", ...)
            
            # Append assistant message with tool_calls + tool result message
            messages.append(response_message)
            messages.append(tool_result_message)
    
    # Exhausted tool calls — force the LLM to produce a draft without tools
    response = await llm_client.chat(messages, tools=None)
    return parse_draft(response.content)
```

**Draft output format:** When the ReAct agent finishes (returns text without tool calls), it must return a JSON object:

```json
{
  "candidates": [
    {
      "title": "Movie Title",
      "year": 2021,
      "tmdb_id": 12345,
      "genres": ["Action", "Thriller"],
      "overview": "Brief plot summary...",
      "rationale": "Why this matches the user's taste"
    }
  ],
  "is_clarification": false,
  "clarification_question": null
}
```

If `ask_user_clarification` was invoked, `is_clarification` is true and `clarification_question` contains the question.

Note: this draft intentionally has no `availability` field. Candidates from `rag_search` carry a `tmdb_id` straight from the RAG metadata, but availability is not known yet at this stage; candidates from `tmdb_fallback_search` are already availability-filtered by construction. Either way, availability is confirmed once, downstream, by the Reflection stage's `verify_recommendation` tool — avoiding a redundant TMDB availability lookup for every candidate.

### Stage 2 — Reflection Agent (`agent/reflection_agent.py`)

**One LLM call** that uses function calling to access the `verify_recommendation` tool. Unlike ReAct, this is a simpler loop — typically one or two tool calls (verify the candidates) then a final verdict.

**System prompt direction:** "You are CineMatch's quality critic. You receive a draft list of movie recommendations and the user's original preferences. Your job is to verify quality, not to search for new movies. Use the verify_recommendation tool to ground your critique in real data — do not just re-read the draft. Check: (1) Does each pick genuinely match the stated mood/taste, not just genre tags? (2) Did anything from the exclude/seen list slip through? (3) Is availability confirmed for the exact country + platform? (4) Is there enough novelty — are these all top-100 IMDB movies, or is there genuine diversity? After verification, return your verdict as JSON: decision (approve/reject/clarify), and the associated payload."

**Tools available to Reflection:**

1. **`verify_recommendation`** — see §7
2. **`ask_user_clarification`** — see §7

**Verdict output format:**

```json
{
  "decision": "approve",
  "final_response": "The formatted recommendation text for the user",
  "critique": null,
  "use_fallback": false,
  "question": null
}
```

Or for reject:

```json
{
  "decision": "reject",
  "final_response": null,
  "critique": "Specific feedback: candidate X doesn't match the mood because... candidate Y isn't available on the specified platform... try using tmdb_fallback_search to find something more obscure.",
  "use_fallback": true,
  "question": null
}
```

Or for clarify:

```json
{
  "decision": "clarify",
  "final_response": null,
  "critique": null,
  "use_fallback": false,
  "question": "I found several action thrillers but I'm not sure about your preference — do you want something more grounded/realistic like Sicario, or more stylized/over-the-top like John Wick?"
}
```

When the decision is "approve", the `final_response` field contains the complete user-facing response text. The Reflection agent composes this directly — there is no separate "composer" LLM call. This saves one LLM call per run.

**Final response format (what the user sees):**

Each recommendation in the response follows this pattern:

```
**Movie Title (Year)** — A 1–2 sentence rationale explaining why this matches what the user asked for, referencing specific elements of their taste/mood/preferences.
✓ Available on [Platform] — [Country]
```

Multiple recommendations are separated by blank lines. The agent decides how many to return (1–5) based on confidence and the nature of the request.

## 7. Backend — Tool Specifications

Each tool lives in `agent/tools/` and exposes two things: (1) a function-calling schema (dict matching OpenAI's tool format) and (2) an async `execute` function.

### `rag_search`

**Schema inputs:** `query_text` (string — the semantic search text, derived from mood/vibe and similar-to titles), `genres` (list of strings, optional — metadata filter), `year_min` (int, optional), `year_max` (int, optional), `min_score` (float, optional), `top_k` (int, default from config).

**Implementation:**
1. Embed `query_text` using the embedding model (`text-embedding-3-small` via LLMod.ai).
2. Build a Pinecone metadata filter combining whichever optional fields were provided. Use `$in` for genre (matches if the movie has any of the requested genres). Use `$gte`/`$lte` for year and score ranges.
3. Query Pinecone with the embedding vector + metadata filter, returning `top_k` results.
4. Return the results: list of `{title, year, tmdb_id, genres, score, overview, pinecone_score}`.

**No TMDB API call** — this tool only reads from Pinecone. The RAG metadata already includes each movie's `tmdb_id` (see §9), so no separate ID-resolution step against the TMDB API is needed. This tool does not check availability — availability is verified downstream by the Reflection stage's `verify_recommendation` tool.

### `tmdb_fallback_search`

**Schema inputs:** `genres` (list of TMDB genre IDs), `year_min` (int, optional), `year_max` (int, optional), `min_rating` (float, optional), `country` (string — ISO 3166-1 code), `platforms` (list of TMDB provider IDs), `max_results` (int, default 10).

**Implementation:**
1. Call TMDB `/discover/movie` with the structured filters: `with_genres`, `primary_release_date.gte/lte`, `vote_average.gte`, `watch_region={country}`, `with_watch_providers={provider_ids joined by |}`.
2. Return: list of `{title, year, tmdb_id, genres, overview, rating, available: true, platform, country}`.

This tool returns movies that are **already filtered by availability** (TMDB discover does this natively), so no separate availability check is needed.

### `verify_recommendation`

**Schema inputs:** `candidates` (array of `{tmdb_id: int, title: string}`), `country` (string), `platforms` (list of strings), `user_mood` (string — the original mood/taste description for comparison).

**Implementation:**
1. For each candidate, **in parallel**:
   a. Fetch TMDB `/movie/{tmdb_id}` for genre, keywords, overview, rating, popularity.
   b. Fetch TMDB `/movie/{tmdb_id}/watch/providers` for current availability in the specified country.
2. Return per candidate: `{tmdb_id, title, available: bool, platform: string|null, genres: list, overview: string, popularity: float, keyword_tags: list, verdict: "pass"|"fail", reason: string}`.

The `verdict` and `reason` are determined by the tool itself (not the LLM): `fail` if availability is not confirmed on the requested platform/country. The LLM (Reflection) uses the rest of the data (genres, overview, popularity, keywords) to assess taste match and novelty — that's the LLM's job, not the tool's.

### `ask_user_clarification`

**Schema inputs:** `question` (string — the clarifying question to ask the user).

**Implementation:** No external API call. The tool simply returns a signal object `{action: "clarify", question: "..."}` that the orchestrator catches. When the orchestrator sees this, it short-circuits and returns the question as the pipeline's response for this turn. The user's next message (with `conversation_history` carrying the prior turns) triggers a fresh pipeline run from Stage 0.

## 9. Data Schemas

### Pinecone Index Schema (already populated — do NOT build an ingestion script)

The Pinecone index `cinematch-movies` is assumed to already exist and be populated. Building or updating this dataset is out of scope for this spec — treat it as a given. Each vector represents one movie:

- **Vector:** 1536 dimensions (output of `text-embedding-3-small` on `"{title}. {overview}"`). Genre is deliberately NOT in the embedded text.
- **Metadata fields:**
  - `title` (string)
  - `year` (integer — release year)
  - `tmdb_id` (integer — the movie's TMDB ID, already resolved at dataset build time; every row has one)
  - `genres` (list of strings, e.g. `["Action", "Thriller"]`)
  - `score` (float — IMDB rating)
  - `overview` (string — plot summary)
  - `budget` (float — may be 0 for missing data)
  - `revenue` (float — may be 0 for missing data)
  - `original_language` (string)

Because `tmdb_id` is already present in the metadata, the agent never needs to resolve a title/year to a TMDB ID at runtime — that lookup has already been done when the dataset was built.

Note: **runtime/duration is NOT in this dataset.** Duration constraints can only be honored via `tmdb_fallback_search`.

## 12. Error Handling

- **Tool call validation:** Before executing any tool call from the ReAct or Reflection agents, validate the tool name against the allowed set and validate argument types against the expected schema. If invalid, return an error string as the tool result (not an exception) so the LLM can self-correct on the next cycle. Cap consecutive invalid tool calls at 3 — if the LLM fails 3 times in a row, abort with a graceful error.
- **TMDB API errors:** If TMDB returns an error or times out on a specific movie, skip that candidate and continue with the rest. If TMDB is completely unreachable, return an error response: `{"status": "error", "error": "Movie database (TMDB) is currently unavailable. Please try again later.", ...}`.
- **Pinecone errors / zero results:** If Pinecone returns zero results (filter too narrow), the ReAct agent should notice this from the empty tool result and adapt — try a broader search, or if feedback allows, use `tmdb_fallback_search`. If Pinecone is unreachable, the orchestrator should catch this and allow the agent to fall back to `tmdb_fallback_search` exclusively (set `use_fallback=True` and re-run ReAct).
- **LLM errors:** Wrap all LLM calls in try/except. Retry once on transient errors (timeout, 5xx). On persistent failure, return an error response.
- **LLM output parsing errors:** If the LLM returns text that doesn't parse as the expected JSON (for TasteExtractor output, ReAct draft, Reflection verdict), retry the call once with a nudge message ("Your previous response was not valid JSON. Please return valid JSON matching the specified format."). On second failure, abort with error.

## 13. Steps Logging

Every LLM call and every tool execution produces a step object appended to the `steps` list. The orchestrator passes this list by reference through all stages. Each function that makes an LLM call or executes a tool is responsible for appending its own step.

Create a helper:

```python
def log_step(steps: list, module: str, system_prompt: str | None, user_prompt: str, response: dict):
    steps.append({
        "module": module,
        "prompt": {
            "system_prompt": system_prompt,
            "user_prompt": user_prompt
        },
        "response": response
    })
```

For LLM calls: `module` is the agent name (e.g. `"TasteExtractor"`, `"ReActAgent"`, `"ReflectionAgent"`), `system_prompt` and `user_prompt` are the actual prompts sent, `response` is the parsed LLM output.

For tool executions: `module` is `"{AgentName}/{tool_name}"` (e.g. `"ReActAgent/rag_search"`), `system_prompt` is `null`, `user_prompt` is the tool input (JSON string), `response` is the tool output.

## 14. Frontend (Deferred)

The frontend/GUI is **not** part of this spec and should not be built now. This document covers the backend only: architecture, agent pipeline, tools, and API endpoints.

The frontend will be designed and built in a later phase, once the backend above is working. When it is built, it must still meet the project's baseline requirements (a simple page at the root URL with a prompt input, a "Run Agent" button calling `POST /api/execute`, a display of the final `response`, a display of the full `steps` trace, and no authentication/login guards) — but the specifics (framework, layout, components, styling) are intentionally left open for that later spec.

Until the frontend exists, the backend should be fully usable and testable on its own (e.g. via `curl`/Postman/the FastAPI docs) — do not build any placeholder UI as part of this phase.

## 17. What NOT to Build

- **No data ingestion script.** The Pinecone index (including each movie's `tmdb_id`) is assumed to already exist and be populated. The ingestion pipeline is a separate project.
- **No TMDB ID resolution by title/year.** Every candidate already has a `tmdb_id` — from RAG metadata or from `tmdb_fallback_search` — so no title/year search-and-match step is needed.
- **No TMDB response caching.** All TMDB calls are live.
- **No frontend/GUI in this phase.** Build the backend only; the frontend is a separate, later spec (see §14).
- **No authentication / user accounts.** When the GUI is eventually built, it must be open access.
- **No TV show support.** Movies only.
- **No email or notification features.**
- **No payment or subscription logic.**
